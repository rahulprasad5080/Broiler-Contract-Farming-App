1. What Changed In This Verified Handoff

The old fixed enums for CatalogItemType, PurchaseType, ExpenseCategoryCode, and TreatmentKind are now string-backed master data. Frontend must load dropdowns from /master-data/type-options or /master-data/type-options/dropdown instead of hardcoding those lists.

Vendor and trader split is now enforced. Vendors feed purchases, vendor payments, inventory vendor views, and vendor ledgers. Traders feed sales, sale receipts, and trader ledgers.

New partner ledger report APIs exist for vendor and trader statement views: /reports/vendors/{vendorId}/ledger and /reports/traders/{traderId}/ledger.

Existing list APIs gained partner-aware filters: purchases can filter by vendorId, payments can filter by vendorId / traderId / referenceType, and inventory ledger can filter by vendorId.

Authentication now includes PATCH /auth/fcm-token for mobile push registration. The previous handoff missed this route.

Coverage in this handoff is verified against the current backend contract: 89 versioned OpenAPI routes plus 4 non-versioned support routes.

5. Core Frontend Flows

Session bootstrap and push registration

Authenticate with POST /api/v1/auth/login or POST /api/v1/auth/login-pin.

Immediately hydrate the signed-in user with GET /api/v1/auth/me and use the returned role plus permissions to gate screens and actions.

After device token acquisition, register or clear push tokens through PATCH /api/v1/auth/fcm-token.

Use POST /api/v1/auth/refresh for silent renewal. Logout uses POST /api/v1/auth/logout with the refresh token body.

Admin setup order

Owner creates the organization through POST /api/v1/auth/register-owner or signs in through the existing account flow.

Load organization settings with GET /api/v1/settings/organization before building finance, approval, or payout setup screens.

Create users, farms, vendors, traders, and catalog items in that order so downstream batch and finance screens have their lookup data available.

Dynamic type masters and dropdown flow

Use GET /api/v1/master-data/type-options/dropdown?category=...&search=... for searchable dropdowns.

Use GET /api/v1/master-data/type-options for fuller admin listings with pagination and includeInactive support.

Create org-specific custom values with POST /api/v1/master-data/type-options and update them with PUT /api/v1/master-data/type-options/{optionId}.

Entries store plain strings, so renaming or deactivating a type option does not rewrite historical rows.

Partner master and statement flow

Use vendor master endpoints for purchases, payment made, inventory vendor filtering, and vendor ledgers.

Use trader master endpoints for sales, payment received, and trader ledgers.

Frontend should send vendorId or traderId whenever a flow supports it. The backend can normalize partyName from the chosen partner or reference row.

Batch lifecycle and operations capture

Create the batch first, then use daily logs, treatments, expenses, comments, and sales under that batch.

Daily logs reject future dates and duplicate logDate entries for the same batch, so the client should pre-check before submit.

Treatments and expenses now use dynamic string values backed by type masters instead of hardcoded enums.

The /batches/{batchId}/costs routes remain a legacy alias of /expenses and should not be used for new frontend work.

Purchase -> inventory -> payment made

Create purchases with POST /api/v1/finance/purchases using vendorId and purchaseType from the new masters.

Non-CHICKS purchases with catalogItemId and quantity create stock movement automatically. CHICKS purchases stay financial only.

Use GET /api/v1/inventory/ledger and GET /api/v1/finance/purchases?vendorId=... for vendor-linked stock and procurement history views.

Record payment made with POST /api/v1/finance/payments. Reference-linked purchase or expense payments must be OUTBOUND and vendor-aligned.

Sale -> payment received

Create sales from the batch screen with POST /api/v1/batches/{batchId}/sales and finalize with PATCH /api/v1/batches/{batchId}/sales/{saleId}/finalize when needed.

Sales belong to traders only. Payment receipts must be INBOUND and trader-aligned.

Use GET /api/v1/finance/payments?traderId=... or GET /api/v1/reports/traders/{traderId}/ledger for trader collection views.

Settlement -> payout -> batch closure

GET /api/v1/batches/{batchId}/settlement loads the current settlement state. POST /api/v1/batches/{batchId}/settlement is an upsert, not a create-only route.

When settlement payment is recorded through POST /api/v1/finance/payments with referenceType=settlement, the backend recomputes payment status and may move the batch to CLOSED automatically once fully paid.

Use GET /api/v1/reports/settlements for list views and the batch settlement route for detail / edit screens.

Reports, ledgers, and exports

Operational and finance summaries should use report endpoints directly; do not recompute P&L, settlement pending amounts, or ledger balances on the client.

Vendor and trader ledgers support dateFrom and dateTo and return openingBalance plus running balance rows for statement-style screens.

Excel and PDF export endpoints return binary files; frontend should call them as blob downloads and preserve the Content-Disposition filename.

6. Frontend-Critical Backend Behaviors

Behavior

Why frontend must care

Server-side visibility is the source of truth.

FARMER and SUPERVISOR scoping is applied by the backend across farms, batches, notifications, dashboards, and most reports. Do not rely on client-side filtering alone.

Closed supervisor batches are available by default.

GET /api/v1/batches includes closed supervisor-visible batches unless the frontend passes a specific status query filter.

Type masters are lookup-only.

Catalog items, purchases, expenses, and treatments store raw strings. There is intentionally no relational foreign key to the type master rows.

Historical rows are stable.

Updating a custom type option changes future dropdown choices only. Existing rows keep the original stored string value.

Partner direction rules are enforced.

Purchase and expense payments must be OUTBOUND, sale receipts must be INBOUND, and reference-linked payments must match the partner on the referenced row.

partyName is not the primary source of truth anymore.

When vendorId, traderId, or a purchase / expense / sale reference is supplied, the backend resolves the final party name from the master or referenced record.

Inventory behavior depends on purchase type.

Purchase type CHICKS does not create stock movement. Feed, medicine, vaccine, equipment, and similar itemized purchases can create stock movement when catalogItemId and quantity are supplied.

Settlement is computed server-side.

Base quantity, growing charges, net payable, paidAmount, pendingAmount, and batch close behavior are backend-derived values.

Vendor ledger includes batch-level chick supply charges.

Vendor statement screens should expect rows coming from batch vendor charges in addition to purchases, expenses, and payments.

Notification generation is mostly backend-driven.

Frontend only lists and marks notifications. Low stock, payment due, approval, and settlement notifications are created by server workflows.

10. Verified API Gaps And No-Route Behaviors

There are no delete endpoints for users, farms, batches, vendors, traders, catalog items, type options, purchases, payments, comments, treatments, sales, or settlements. Design edit / deactivate UX rather than delete-first UX.

Type options do not have a delete route. Use PUT /api/v1/master-data/type-options/{optionId} with isActive=false to retire a custom option.

There is no update or delete API for treatment rows. Treatments are create + list only in the current backend.

Comments are create + list only. There is no edit or delete endpoint for comments in the current API set.

Sales do not have a general update route after creation. The supported follow-up action is PATCH /batches/{batchId}/sales/{saleId}/finalize.

There is no file-upload API in src/controllers. Frontend must upload externally and then send URLs such as attachmentUrl, billPhotoUrl, billPhotoUrl, proofUrl, or billPhotoUrl to backend payloads.

Subscription approval / review APIs are not exposed in the current frontend-facing backend routes. The owner can request plans and submit payment proof, but approval is handled outside the current route set.

Notification test routes are diagnostic utilities, not business-user flows. Keep them out of normal production navigation.
